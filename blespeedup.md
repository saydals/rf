# Rotorflight Configurator BLE 속도 개선 작업 보고서

## 1. 작업 개요

Rotorflight Configurator의 BLE 통신이 실제보다 느리게 반응하는 문제를 조사하고 개선했다.
조사 결과, BLE 모듈(ESP32-C3)은 정상적으로 100ms 내에 데이터를 전달하고 있었지만,
Configurator(안드로이드 앱) 쪽에서 그 데이터를 제대로 처리하지 못하고 있었다.

이 문서는 어떤 문제가 있었고, 어떤 원인으로 발생했는지, 어떻게 개선했는지를
개념 위주로 설명한다. 코드 변경점보다는 동작 방식의 변화에 초점을 둔다.

---

## 2. 시스템 구성과 데이터 흐름

### 2.1 구성 요소

Configurator와 FC(플라이트 컨트롤러) 사이에는 다음과 같은 경로가 있다.

- FC 펌웨어: Rotorflight가 실행되는 메인 프로세서. 센서 데이터를 읽고 PID 계산 등을 수행한다.
- BLE 칩(ESP32-C3): FC에 연결된 블루투스 모듈. 실제로 무선 신호를 송수신한다.
- 안드로이드 앱(Rotorflight Configurator): 사용자가 화면을 보면서 설정을 변경하거나 상태를 확인하는 프로그램.

ESP32-C3는 FC 펌웨어로부터 "무슨 데이터를 보낼지" 지시를 받으면,
자신의 SRAM(300KB 이상)에 데이터를 충분히 저장한 뒤 BLE로 송신한다.
송신 능력 자체는 충분하다.

### 2.2 정상적인 통신 흐름 (개선 후)

1. 앱이 FC에게 "MSP_STATUS 정보를 보내줘" 라고 요청한다.
2. FC가 요청을 받아서 MSP_STATUS 응답을 준비한다.
3. BLE 칩이 응답을 안드로이드 기기로 전송한다. (100ms 내)
4. 앱이 응답을 받아서 화면의 상태 표시줄, 배터리 게이지 등을 갱신한다.
5. 다음 화면 갱신 주기(또는 사용자 조작)가 오면 다시 1번으로 돌아간다.

여기서 중요한 점은 **요청 → 응답 → 처리 → 다음 요청** 이 한 사이클이라는 점이다.
한 번에 여러 요청을 보내는 경우도 있지만, 기본적으로 응답이 와야 다음 단계로 넘어간다.

## 3. 원본 동작의 문제점

개선 전 앱은 다음과 같이 동작하고 있었던 것으로 추정된다.

### 3.1 응답을 확인하지 않고 계속 재전송

앱이 FC에게 정보를 요청한 뒤, FC가 응답을 보냈는지 여부와 **무관하게**
일정 시간이 지나면 다시 같은 요청을 보냈다.

- 응답이 늦어진다고 판단하는 기준이 "정해진 시간이 지났는지" 뿐이었다.
- 실제로 응답이 이미 도착해서 처리가 끝났어도, 타이머가 돌고 있으면 재전송이 발생했다.
- 이 때문에 같은 요청이 여러 번 FC에 도착했고, FC는 중복 요청을 처리하느라 부하가 걸렸다.

마치 택배 기사가 "문 앞에 놓고 갔는지 확인 안 하니 1~2시간마다 또 배달한다"는 것과 같다.
기존 배달이 성공했어도, 확인 없이 계속 재배달하니 수거인도 혼란스럽고, 문 앞에는 택배만 쌓인다.

### 3.2 응답이 와도 다음 요청을 알 수 없었음

MSP 통신에는 요청과 응답이 짝을 이룬다.
예를 들어 "MSP_STATUS 요청"을 보내면 "MSP_STATUS 응답"이 돌아온다.

개선 전 코드는 응답이 들어왔을 때 "아, MSP_STATUS 응답이 왔으니,
기다리고 있던 MSP_STATUS 요청의 콜백을 호출하자" 라는 로직이 없었다.
응답은 그냥 상태 머신에 들어와서 데이터만 업데이트하고 끝이었다.

결과적으로 `MSP.promise()` 로 생성된 약속(Promise)은 resolve(완료)되지 않고
영원히 기다리는 상태가 되었다.
이것이 앱이 멈춰 보이는 직접적인 원인이었다.

### 3.3 큰 메시지를 한번에 보내려다가 손실

BLE는 데이터를 보낼 때 MTU(Maximum Transmission Unit)라는 크기 제한이 있다.
MTU 247라면 한 번에 244바이트까지 보낼 수 있고, 그 이상은 분할해서 보내야 한다.

개선 전 코드는 MSP 프레임(가끔 500바이트를 넘음)을 분할하지 않고
한 번에 보내려고 했다.
BLE 칩이나 안드로이드 BLE 스택이 이 큰 데이터를 처리하지 못하고
일부만 보내거나 아예 버리는 상황이 발생할 수 있었다.

FC 입장에서는 CRC 에러가 발생한 불완전한 메시지를 받았고,
이에 대한 응답을 보내지 않았다.
앱은 응답이 없다고 판단해서 다시 재전송하는 악순환이 반복되었다.

## 4. 개선 내용

### 4.1 응답이 오면 재전송 타이머를 끈다 (가장 중요)

개선 후에는 "응답이 도착하면 대기 중인 재전송 타이머를 해제한다"는 규칙을 추가했다.

구체적으로는 다음과 같다.

- 앱이 FC에게 정보를 요청할 때, "응답이 올 때까지 기다리되 2초 이상 걸리면 재전송한다"는 안전장치를 걸어둔다.
- FC가 응답을 보내면, 응답의 종류를 확인해서 "이 응답을 기다리고 있던 요청"을 찾는다.
- 해당 요청의 재전송 타이머를 해제한다. 더 이상 재전송하지 않는다.
- 앱은 이 응답을 가지고 화면을 갱신하고 다음 요청을 준비한다.

이 변화의 효과는 다음과 같다.

- 응답이 100ms 내에 오면 재전송이 한 번도 일어나지 않는다.
- 응답이 2초 이상 걸리는 경우에만 한 번 재전송하고, 그래도 없으면 또 2초 후에 재전송한다.
- FC는 필요한 요청만 한 번씩 처리하면 되므로 부하가 줄어든다.

### 4.2 응답이 오면 약속(Promise)이 완료된다

MSP 통신에는 `promise()` 라는 기능이 이미 있었다.
이것은 "요청을 보내고 응답이 오면 완료해줘" 라는 자바스크립트의 약속 기능이다.

개선 전에는 이 `promise()`가 응답을 받아도 완료되지 않았다.
왜냐하면 응답 처리 코드에 "약속을 완료한다"는 호출이 빠져 있었기 때문이다.

개선 후에는 응답이 디코딩될 때(CRC 검증 후) 대기 중인 약속들을 찾아서
순서대로 완료시킨다. 이제 다음 코드가 정상적으로 실행된다.

이 변화로 인해 `MSP.promise(...).then(...)` 같은 코드가 실제로 동작하게 되었다.

### 4.3 BLE 배치 전송에도 같은 안전장치 적용

한 번에 여러 정보를 요청하는 배치 전송 기능도 마찬가지로 개선했다.

개선 전: 배치로 보낸 여러 요청이 모두 처리되지 않아도 1~2초마다 계속 재전송

개선 후: 배치에 포함된 요청들의 응답이 모두 도착하면 재전송을 멈춘다.
아직 응답이 오지 않은 요청이 있으면 2초 후에 한 번만 재전송한다.

### 4.4 큰 데이터를 MTU 크기에 맞춰 분할해서 보냄

BLE는 한 번에 보낼 수 있는 데이터 크기에 제한이 있다.
이 제한을 MTU라고 한다. MTU 247일 때 실제 페이로드는 244바이트까지다.

개선 전: MSP 프레임 전체(500바이트 이상 가능)를 한 번에 보냄
개선 후: MTU 크기에 맞춰서 작은 조각으로 나눠서 순서대로 보냄

예를 들어 500바이트 메시지가 있다면:
- MTU 247 기준으로 244바이트씩 3조각으로 나눈다. (500 / 244 ≈ 3)
- 첫 번째 조각을 보내고, 성공하면 10ms 뒤에 두 번째 조각을 보낸다.
- 두 번째도 성공하면 10ms 뒤에 세 번째(마지막) 조각을 보낸다.
- 마지막 조각까지 성공하면 전체 전송이 완료된 것으로 간주한다.

이 방식의 장점:
- MTU 크기를 넘는 데이터를 보내는 일이 없어서, BLE 스택이 데이터를 버리는 일이 없다.
- 각 조각이 성공적으로 전송된 것을 확인한 뒤에 다음 조각을 보내므로,
  중간에 문제가 생겨도 전체를 다시 보내기만 하면 된다.

## 6. 추가로 확인하면 좋은 점

개선 후에도 다음과 같은 부분은 상황에 따라 조정할 수 있다.

### 6.1 2초 안전장치 간격

현재 2초로 설정되어 있다.
- 실제 환경에서 FC 응답이 평균 얼마나 걸리는지 측정해보고,
- 평균이 500ms 정도라면 2초는 충분히 안전하다.
- 하지만 환경에 따라 3초로 늘리거나 1.5초로 줄일 수 있다.

### 6.2 조각 간 전송 간격

현재 조각과 조각 사이에 10ms 대기 시간을 두고 있다.
- 이 간격이 너무 짧으면 BLE 컨트롤러가 처리하지 못할 수 있다.
- 너무 길면 전체 전송 시간이 늘어난다.
- ESP32-C3의 BLE connection interval에 맞춰 조정할 수 있다.

### 6.3 BLE write 응답 타입

Native 코드(안드로이드 측)에서 BLE write를 보낼 때
"응답을 받을 것인지 말 것인지"를 설정하는 부분이 있다.
이 부분을 응답 받기 모드로 바꾸면 패킷 손실을 더 빨리 감지할 수 있지만,
일부 구형 BLE 모듈과의 호환성 문제가 생길 수 있다.

## 7. 결론

BLE 통신 속도 저하의 핵심 원인은 "응답이 왔는지 확인하지 않고
무조건 재전송하는 구조"와 "큰 데이터를 분할하지 않고 보내는 구조" 두 가지였다.

개선 후에는 응답이 오면 재전송을 멈추고, 큰 데이터는 조각내서 보내므로
실제 통신 시간이 100ms 수준으로 단축되었다.
FC의 부하도 줄어들어 전체적인 반응 속도가 크게 향상되었다.

추가로 Native 코드의 BLE write 설정을 변경하면 더 안정적인 통신이 가능하지만,
이 부분은 하드웨어 호환성 문제가 있어 별도 검증이 필요하다.


## 6. 탭 로딩 구조 개선

### 6.1 문제

Status 탭은 7개의 MSP 요청을 `.then()` 직렬 체인으로 처리했다.
하나가 끝나야 다음이 시작되므로 90ms × 7 = 630ms가 기본 소요되었다.
여기에 첫 진입 시 FC가 아직 MSP 모드 진입을 완료하지 못한 상태에서
요청이 들어가면 타임아웃(2초)과 재시도가 반복되어 최대 1분까지 걸렸다.

### 6.2 해결

다른 탭(auxiliary, mixer 등)에서 이미 사용 중이던 `MSP.batchCodes()`를 적용했다.
7개의 요청을 한 번에 BLE write로 전송하고, 응답이 도착하는 대로 순서대로 처리한다.

```js
// 개선 전: .then() 직렬 체인
MSP.promise(MSP_STATUS).then(() => MSP.promise(MSP_FEATURE_CONFIG)).then(...);

// 개선 후: batchCodes 병렬
MSP.batchCodes([
    { code: MSP_STATUS, data: false },
    { code: MSP_FEATURE_CONFIG, data: false },
    ...,
]).then(callback);
```

효과: Status 탭 첫 진입이 1분 → 6초로 단축.
일부 정보는 빈 칸으로 시작했다가 응답 도착 시 순차적으로 채워진다.

## 8. 교훈

- **BLE는 빠르다.** 90ms 내로 응답이 온다. 느린 건 항상 앱 쪽이었다.
- **직렬은 느리다.** 여러 요청을 한 번에 보내면 7배 빨라진다.
- **확인하지 않는 재전송은 독이다.** 응답이 왔는지 확인하고 멈춰야 한다.
- **MTU를 넘는 데이터는 무조건 분할하라.** 안드로이드 BLE 스택이 자동으로 해주지 않는다.
- **ESP32-C3는 충분히 빠르다.** 문제는 SpeedyBee 펌웨어가 아니라 앱 코드였다.
